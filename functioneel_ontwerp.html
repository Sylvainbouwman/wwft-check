1. Aanleiding en doelstelling

De WWFT (Wet ter voorkoming van witwassen en financieren van terrorisme) verplicht poortwachtersorganisaties tot het uitvoeren van cliëntenonderzoek (CDD – Client Due Diligence) vóór en gedurende de dienstverlening. Bij niet-naleving riskeert de organisatie boetes van de AFM/DNB en reputatieschade.

Op dit moment voeren medewerkers dit onderzoek handmatig uit via publieke bronnen (Google, KvK.nl, diverse sanctielijst-websites). Dit is tijdrovend, foutgevoelig en produceert geen gestandaardiseerd en auditeerbaar dossier.

Doelstelling: Een interne webapplicatie bouwen die medewerkers stap voor stap begeleidt door het WWFT cliëntenonderzoek, automatisch relevante bronnen raadpleegt, een risicobeoordeling uitvoert en een auditeerbaar rapport genereert.

2. Huidige situatie

Hoe werkt het nu:

  • Medewerker zoekt handmatig op via kvk.nl en noteert gegevens in een Word/Excel-bestand.
  • Sanctielijsten worden handmatig geraadpleegd (EU-website, diverse Google-zoekopdrachten).
  • UBO's worden opgevraagd via kvk.nl/ubo-register (handmatig).
  • Risicobeoordeling is niet gestandaardiseerd; verschilt per medewerker.
  • Geen centrale opslag van dossiers; geen herinneringen voor periodieke hertoetsing.
  • Geen audit trail van wie welk onderzoek heeft gedaan en wanneer.
KnelpuntImpact
Tijdrovend (15–45 min per cliënt)Hoog – schaalt niet bij groei
Niet reproduceerbaar / gestandaardiseerdHoog – compliancerisico
Geen audit trailHoog – problematisch bij toezichthouder
Geen periodieke hertoetsing-signaleringMidden – risico op verouderde dossiers
Kennis afhankelijk van individuMidden – kwetsbaar bij personeelswissels

3. Gewenste situatie

Een webapplicatie die medewerkers begeleidt door het gehele CDD-proces:

  1. Cliënt opzoeken via KvK-nummer → automatisch bedrijfsgegevens ophalen
  2. UBO identificeren via KvK UBO Register API (separate koppeling)
  3. Sanctie- en PEP-screening op naam van bedrijf en UBO's via geautomatiseerde lijsten
  4. Risicobeoordeling op basis van sector (SBI-code), rechtsvorm, land en screeningsuitslag
  5. Dossiervorming met documentupload (ID-bewijs, UBO-verklaring)
  6. Auditeerbaar rapport met tijdstempel en medewerkersnaam
  7. Hertoetsing-signalering: automatische melding bij verlopen dossiers
Tijdwinst inschatting: Van 15–45 minuten handmatig naar ~5 minuten gestructureerd per cliënt. Bij 100 nieuwe cliënten per jaar: ~50–65 uur tijdsbesparing.

5. Use Cases

UC01 – Nieuw cliëntdossier aanmaken

ActorMedewerker
TriggerNieuwe cliëntrelatie of opdracht wordt aangegaan
PreconditiesMedewerker is ingelogd; KvK-nummer of naam van cliënt is bekend
PostconditiesDossier aangemaakt in systeem; eerste screening uitgevoerd
Hoofdscenario
  1. Medewerker klikt op "Nieuw cliëntenonderzoek"
  2. Voert KvK-nummer in
  3. Systeem haalt bedrijfsgegevens op (UC02)
  4. Medewerker verifieert gegevens en start screening (UC04)
  5. Risicobeoordeling wordt uitgevoerd (UC05)
  6. Rapport wordt gegenereerd (UC06)
  7. Dossier opgeslagen met tijdstempel en medewerkersnaam
AlternatiefParticulier (natuurlijk persoon): handmatige invoer naam + geboortedatum

UC02 – KvK-gegevens automatisch ophalen

ActorSysteem (geautomatiseerd)
TriggerMedewerker voert KvK-nummer in
PreconditiesGeldige KvK API-verbinding beschikbaar
PostconditiesBedrijfsnaam, adres, rechtsvorm, SBI-code(s), datum oprichting ingevuld
Hoofdscenario
  1. Systeem verstuurt API-verzoek naar KvK Handelsregister
  2. Basisgegevens worden opgehaald en getoond
  3. SBI-code wordt automatisch gelabeld als risico-indicator
  4. Medewerker kan gegevens aanvullen of corrigeren
UitzonderingKvK-nummer niet gevonden → melding aan medewerker

UC03 – UBO identificeren

ActorMedewerker + Systeem
TriggerNa ophalen KvK-basisgegevens
PreconditiesKvK UBO Register API gekoppeld (nog aan te vragen)
PostconditiesUBO('s) geïdentificeerd en opgeslagen in dossier
Hoofdscenario
  1. Systeem haalt UBO-informatie op via KvK UBO API
  2. Namen, geboortedatums en aandeelhouderpercentages worden getoond
  3. UBO's worden automatisch meegenomen in sanctie/PEP-screening (UC04)
Huidig alternatiefHandmatige raadpleging op kvk.nl/ubo-register + handmatige invoer
PrioriteitHoog – WWFT verplicht UBO-identificatie

UC04 – Sanctie- en PEP-screening

ActorSysteem (geautomatiseerd)
TriggerNa UC02 / UC03; of bij hertoetsing
PreconditiesOpenSanctions API bereikbaar
PostconditiesScreeningresultaat vastgelegd; eventuele treffers gemarkeerd
Hoofdscenario
  1. Systeem screent bedrijfsnaam en UBO-namen via OpenSanctions API
  2. Resultaten worden geclassificeerd: geen treffer / mogelijke treffer / sterke treffer
  3. Sterke treffer → automatisch risico HOOG + melding compliance officer
  4. Screeningsresultaat wordt opgeslagen met tijdstempel en versie sanctielijst
UitzonderingAPI niet bereikbaar → melding, handmatige screening vereist

UC05 – Risicobeoordeling uitvoeren

ActorSysteem + Medewerker (eindoordeel)
TriggerNa UC04
PreconditiesBedrijfsgegevens en screeningsresultaten beschikbaar
PostconditiesRisicoclassificatie vastgelegd: Laag / Midden / Hoog
Hoofdscenario
  1. Systeem berekent automatisch risicoscore op basis van:
    • Sector (SBI-code) – bijv. vastgoed, kansspelen = hoog
    • Rechtsvorm – bijv. stichting, coöperatie = verhoogd
    • Land van herkomst / vestiging
    • Screeningsuitslag (sanctielijst/PEP-treffer)
  2. Medewerker kan risicoclassificatie bijstellen met motivatie
  3. Bij HOOG risico: melding compliance officer + verplicht verscherpt onderzoek (art. 8 WWFT)

UC06 – Rapport genereren

ActorMedewerker
TriggerNa afronding cliëntenonderzoek
PostconditiesPDF/tekstrapport opgeslagen in dossier, downloadbaar
Hoofdscenario
  1. Systeem compileert alle stappen in gestructureerd rapport
  2. Rapport bevat: cliëntgegevens, UBO's, screeningresultaten, risicoclassificatie, motivatie, datum, medewerker
  3. Rapport wordt opgeslagen als auditeerbaar bewijs
  4. Medewerker kan rapport downloaden (PDF / Word)

UC07 – Periodieke hertoetsing

ActorSysteem + Medewerker
TriggerTijdsinterval verstreken (bijv. 1 jaar) of wijziging bij cliënt
PostconditiesNieuw screeningmoment vastgelegd; dossier actueel
Hoofdscenario
  1. Systeem signaleert medewerker dat hertoetsing vereist is
  2. Medewerker herhaalt screening (UC04) en risicobeoordeling (UC05)
  3. Nieuw rapport gegenereerd en aan dossier toegevoegd

6. API's en koppelingen

Onderstaand overzicht toont welke koppelingen benodigd zijn, wat de status is en wat de verwachte kosten zijn.

Koppeling Wat het levert Status Kostenindicatie Prioriteit
KvK Handelsregister API Bedrijfsnaam, adres, rechtsvorm, SBI-code, datum oprichting Beschikbaar Gratis (na registratie KvK Developer Portal) Hoog
KvK UBO Register API Naam, geboortedatum, percentage aandeelhouderschap van UBO's Aanvragen vereist Gratis (overheidsdata) – aparte toegang aanvragen bij KvK Hoog
OpenSanctions (open source) EU/VN/OFAC sanctielijsten, PEP-registers, ~40 databronnen Beschikbaar Gratis (cloud API beperkt gebruik) / Zelf hosten ~€50/mnd server Hoog
OpenSanctions Cloud API Zelfde als boven, maar hogere limieten en SLA Aan te kopen ~€150–500/mnd afhankelijk van volume Midden
World-Check (Refinitiv/LSEG) Commerciële PEP + sanctie + adverse media database Optioneel €15.000–50.000/jaar (enterprise licentie) Optioneel
Comply Advantage PEP + sanctie + nieuws screening, per-search pricing Optioneel €500–2.000/mnd + per search Optioneel
iDIN / eHerkenning Digitale identiteitsverificatie voor particulieren / eenmanszaken Optioneel €0,30–0,70 per verificatie Fase 2
Conclusie koppelingen: De minimale set voor een werkende WWFT-tool bestaat uit KvK Basisregistratie (al beschikbaar), KvK UBO Register (aanvragen bij KvK, gratis) en OpenSanctions (gratis). Commerciële PEP-databases zijn niet direct vereist, maar verhogen de kwaliteit van de screening.

7. Globale architectuur

┌─────────────────────────────────────────────────────────────┐
│                     WWFT Check Tool                         │
│                                                             │
│  Webbrowser (medewerker)                                    │
│       │                                                     │
│       ▼                                                     │
│  Frontend (React / Streamlit)                               │
│       │                                                     │
│       ▼                                                     │
│  Backend API (Python FastAPI)                               │
│   ├── /api/clients        – dossiers                        │
│   ├── /api/screening      – sanctie check                   │
│   ├── /api/reports        – rapportage                      │
│   └── /api/auth           – inloggen (Azure AD SSO)         │
│       │                                                     │
│       ├──► KvK API         (basisgegevens + UBO)           │
│       ├──► OpenSanctions   (sanctie + PEP screening)        │
│       └──► PostgreSQL      (dossiers, audit trail)          │
│                                                             │
│  Hosting: on-premise / Azure (AVG-compliant)               │
└─────────────────────────────────────────────────────────────┘

Authenticatie: Koppeling met Azure Active Directory (of bestaand SSO-systeem) zodat medewerkers met hun werkaccount inloggen – geen apart wachtwoord nodig.

Gegevensopslag: Alle dossiers worden lokaal opgeslagen (on-premise of AVG-conforme cloud). Persoonsgegevens worden versleuteld opgeslagen. Bewaarplicht conform WWFT: minimaal 5 jaar.
